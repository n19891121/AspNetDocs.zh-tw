## <a name="implementation"></a><span data-ttu-id="bd7ec-130">實作</span><span class="sxs-lookup"><span data-stu-id="bd7ec-130">Implementation</span></span>

<span data-ttu-id="bd7ec-131">Signalr，每則訊息是透過訊息匯流排傳送。</span><span class="sxs-lookup"><span data-stu-id="bd7ec-131">In SignalR, every message is sent through a message bus.</span></span> <span data-ttu-id="bd7ec-132">訊息匯流排實作[IMessageBus](https://msdn.microsoft.com/library/microsoft.aspnet.signalr.messaging.imessagebus(v=vs.100).aspx)介面，可提供發佈/訂閱抽象概念。</span><span class="sxs-lookup"><span data-stu-id="bd7ec-132">A message bus implements the [IMessageBus](https://msdn.microsoft.com/library/microsoft.aspnet.signalr.messaging.imessagebus(v=vs.100).aspx) interface, which provides a publish/subscribe abstraction.</span></span> <span data-ttu-id="bd7ec-133">藉由取代預設的背板運作**IMessageBus**與專為該後擋板匯流排。</span><span class="sxs-lookup"><span data-stu-id="bd7ec-133">The backplanes work by replacing the default **IMessageBus** with a bus designed for that backplane.</span></span> <span data-ttu-id="bd7ec-134">比方說，是適用於 Redis 訊息匯流排[RedisMessageBus](https://msdn.microsoft.com/library/microsoft.aspnet.signalr.redis.redismessagebus(v=vs.100).aspx)，並使用 Redis [pub/sub](http://redis.io/topics/pubsub)機制來傳送和接收訊息。</span><span class="sxs-lookup"><span data-stu-id="bd7ec-134">For example, the message bus for Redis is [RedisMessageBus](https://msdn.microsoft.com/library/microsoft.aspnet.signalr.redis.redismessagebus(v=vs.100).aspx), and it uses the Redis [pub/sub](http://redis.io/topics/pubsub) mechanism to send and receive messages.</span></span>

<span data-ttu-id="bd7ec-135">每個伺服器執行個體連線到透過匯流排後擋板。</span><span class="sxs-lookup"><span data-stu-id="bd7ec-135">Each server instance connects to the backplane through the bus.</span></span> <span data-ttu-id="bd7ec-136">當訊息傳送時，會先移至後擋板，並後擋板將它傳送至每一部伺服器。</span><span class="sxs-lookup"><span data-stu-id="bd7ec-136">When a message is sent, it goes to the backplane, and the backplane sends it to every server.</span></span> <span data-ttu-id="bd7ec-137">當伺服器收到訊息時從後擋板時，它會將訊息放在其本機快取中。</span><span class="sxs-lookup"><span data-stu-id="bd7ec-137">When a server gets a message from the backplane, it puts the message in its local cache.</span></span> <span data-ttu-id="bd7ec-138">伺服器再將訊息傳遞至用戶端從其本機快取。</span><span class="sxs-lookup"><span data-stu-id="bd7ec-138">The server then delivers messages to clients from its local cache.</span></span>

<span data-ttu-id="bd7ec-139">每個用戶端連線，在讀取訊息資料流的用戶端的程序是使用資料指標來追蹤。</span><span class="sxs-lookup"><span data-stu-id="bd7ec-139">For each client connection, the client's progress in reading the message stream is tracked using a cursor.</span></span> <span data-ttu-id="bd7ec-140">（資料指標表示訊息資料流中的位置）。如果用戶端中斷連線，並再重新連接時，它會要求用戶端的資料指標的值之後進入的任何訊息匯流排。</span><span class="sxs-lookup"><span data-stu-id="bd7ec-140">(A cursor represents a position in the message stream.) <span data-ttu-id="bd7ec-143">資料指標機制的運作即使用戶端時，會路由至不同的伺服器上，在重新連線。</span><span class="sxs-lookup"><span data-stu-id="bd7ec-143">The cursor mechanism works even if a client is routed to a different server on reconnect.</span></span> <span data-ttu-id="bd7ec-144">後擋板所知的所有伺服器，並不重要的用戶端連接至的伺服器。</span><span class="sxs-lookup"><span data-stu-id="bd7ec-144">The backplane is aware of all the servers, and it doesn't matter which server a client connects to.</span></span>

## <a name="limitations"></a><span data-ttu-id="bd7ec-145">限制</span><span class="sxs-lookup"><span data-stu-id="bd7ec-145">Limitations</span></span>

<span data-ttu-id="bd7ec-146">使用後擋板，最大訊息輸送量低於時用戶端直接與單一伺服器節點。</span><span class="sxs-lookup"><span data-stu-id="bd7ec-146">Using a backplane, the maximum message throughput is lower than it is when clients talk directly to a single server node.</span></span> <span data-ttu-id="bd7ec-147">這是因為後擋板轉送至每個節點中，每個訊息，讓後擋板成為瓶頸。</span><span class="sxs-lookup"><span data-stu-id="bd7ec-147">That's because the backplane forwards every message to every node, so the backplane can become a bottleneck.</span></span> <span data-ttu-id="bd7ec-148">這項限制是否發生問題，則應用程式而定。</span><span class="sxs-lookup"><span data-stu-id="bd7ec-148">Whether this limitation is a problem depends on the application.</span></span> <span data-ttu-id="bd7ec-149">例如，以下是一些典型的 SignalR 案例：</span><span class="sxs-lookup"><span data-stu-id="bd7ec-149">For example, here are some typical SignalR scenarios:</span></span>

- <span data-ttu-id="bd7ec-150">[伺服器廣播](tutorial-server-broadcast-with-aspnet-signalr.md)（例如，股票行情指示器）：背板適用於此案例中，因為伺服器控制傳送訊息的速率。</span><span class="sxs-lookup"><span data-stu-id="bd7ec-150">[Server broadcast](tutorial-server-broadcast-with-aspnet-signalr.md) (e.g., stock ticker): Backplanes work well for this scenario, because the server controls the rate at which messages are sent.</span></span>
- <span data-ttu-id="bd7ec-151">[用戶端到用戶端](tutorial-getting-started-with-signalr.md)（例如聊天）：在此案例中後, 擋板如果可能會發生瓶頸的訊息數目隨著; 的用戶端數目也就是說，如果訊息的速率成長按比例越多的用戶端加入。</span><span class="sxs-lookup"><span data-stu-id="bd7ec-151">[Client-to-client](tutorial-getting-started-with-signalr.md) (e.g., chat): In this scenario, the backplane might be a bottleneck if the number of messages scales with the number of clients; that is, if the rate of messages grows proportionally as more clients join.</span></span>
- <span data-ttu-id="bd7ec-152">[高頻率即時](tutorial-high-frequency-realtime-with-signalr.md)（例如，即時遊戲）：後擋板不建議此案例中。</span><span class="sxs-lookup"><span data-stu-id="bd7ec-152">[High-frequency realtime](tutorial-high-frequency-realtime-with-signalr.md) (e.g., real-time games): A backplane is not recommended for this scenario.</span></span>
